# B-104 1단계 — 비매물 탭 UI 진단·시안

작성일: 2026-07-19

기준: `origin/master` 77a9f80, 제품 코드 변경 없음
목업 데이터: 액션 24건(미완료 19·완료 5), 수집함 36건, 게스트 자산 4건

## 결론

**추천 1안은 “대시보드·자산은 현행 유지 + 액션은 콤팩트 그룹 행 +
수집함은 갤러리 기본 + 전 탭 내부 스크롤 앱셸”이다.**

- 길이의 주원인은 전체폭·여백이 아니라 액션/수집함의 동일 카드 반복이다.
  B-95의 2열은 높이를 줄였지만 정보 구조는 바꾸지 못했다.
- 대시보드는 현재도 한 화면 요약 역할이 명확하다. 대대적 재편 이득이 작다.
- 자산은 표 편집이 핵심이라 카드화보다 현행 표를 보존하는 편이 안전하다.
- 액션은 정보 손실 없는 콤팩트 행과 명시적 그룹 구획이 가장 빠르게 훑어진다.
- 수집함은 이미 존재하는 검색·필터·정렬·목록/갤러리 전환을 유지하면서
  갤러리를 기본값으로 바꾸는 안이 효과 대비 비용이 가장 낮다.
- 스크롤 통일은 B-53의 `--app-top-h`와 B-98의 실측 패턴을 재사용할 수
  있어 과거 B-94 시점보다 작아졌지만, 모바일 회귀면이 넓어 별도 커밋으로
  분리해야 한다.

최종 채택과 구현 순서는 사용자·커맨드센터가 결정한다.

## 현행 진단

| 탭 | 현행 형태 | 진단 |
|---|---|---|
| 대시보드 | 여정·요약 2장·Top 3 | 데스크톱은 한 화면에 들어오며 목적이 명확하다. 모바일은 자연스러운 1열이므로 현행 유지가 합리적이다. |
| 자산 | 요약 2장 + 스프레드시트형 등록부 | 길이보다 편집 정확성이 중요하다. 실제 항목이 늘면 내부 스크롤 통일 효과가 크다. |
| 액션 | 동일 행 2열 + 완료 구획 | 24건에서 줄 수는 줄어도 “끝없는 체크리스트” 인상이 남는다. 상태·담당·분류가 텍스트 배지에 묻힌다. |
| 수집함 | 상세 카드 2열, 갤러리 선택 가능 | 36건에서 원문·버튼까지 모든 카드에 반복되어 가장 길다. 사진/유형 중심 탐색에는 기존 갤러리가 더 적합하다. |

전제인 매물탭 기준 전체폭·외곽 여백은 모든 안에서 유지했다. 검색·필터·
정렬·갤러리·2열 기능도 제거하지 않고 기본 모드 또는 배치만 바꿨다.

## 탭별 제안

### 대시보드

1. **D1 현행 유지(추천)**

   여정 → 자산/매물 요약 → Top 3의 흐름을 그대로 둔다. 내부 스크롤 앱셸만
   적용한다. 가장 낮은 비용으로 탭 전환 감각만 통일할 수 있다.

2. **D2 콤팩트 커맨드센터**

   데스크톱에서 자산·매물 요약과 Top 3를 같은 행에 놓아 첫 화면 밀도를
   높인다. 모바일은 현행 1열로 돌아간다. 한눈보기는 좋아지지만 대시보드
   자체가 이미 짧아 체감 이득은 작다.

### 자산

1. **A1 현행 표 유지(추천)**

   요약과 등록부 순서를 유지하고 패널 내부 스크롤만 적용한다. 금액 입력,
   붙여넣기, 노트 등 현재 기능과 시선 흐름을 보존한다.

2. **A2 요약/등록부 2구역**

   데스크톱 왼쪽에 요약·노트, 오른쪽에 등록부를 배치한다. 큰 화면 활용은
   좋아지지만 왼쪽 sticky 구역과 Tiptap 노트의 포커스/높이 검증이 필요하다.
   모바일에서는 다시 1열이다.

### 액션

1. **AC1 상태·담당 보드**

   미완료 항목을 담당/상태 칼럼처럼 카드화한다. 책임 소재가 잘 보이지만
   현재 `actions`에는 워크플로 상태가 `done` 두 값뿐이라 진짜 칸반을 만들면
   스키마 확장이 필요하다. 스키마를 늘리지 않는다면 담당자 보드로만 제한해야
   하며, 모바일에서는 칼럼 가로 스크롤 또는 1열 전환이 필요하다.

2. **AC2 콤팩트 밀도 행**

   한 행에 체크·할 일·분류·담당·마감·액션을 정렬하는 테이블형이다.
   세로 효율이 가장 좋고 정보 손실이 없다. 다만 작은 화면에서는 보조 열을
   본문 아래 칩으로 재배치해야 한다.

3. **AC3 명시적 그룹 + 콤팩트 행(추천)**

   “일반/매물준비/계약” 또는 “담당자”를 명시적 섹션으로 보여주고 각 섹션
   안은 AC2 밀도 행을 쓴다. 접기로 숨기지 않고 전체 항목을 계속 노출한다.
   현재 category·assignee 데이터를 그대로 활용하며 모바일 1열 전환도 자연스럽다.

### 수집함

1. **SC1 갤러리 기본화(추천)**

   기존 목록/갤러리 토글, 검색·필터·정렬을 모두 유지하되 첫 진입을 갤러리로
   바꾼다. 사진 없는 카드도 유형색 프리뷰 영역을 사용해 시각적 리듬을 만든다.
   상세 원문과 상태 변경은 카드 클릭 후 기존 편집 모달에서 처리한다.

2. **SC2 masonry 상세 카드**

   원문 길이에 맞춰 카드 높이를 살리는 다단 배치다. 메모를 바로 훑기 좋지만
   읽기 순서가 열 단위로 흐르고, 필터 결과 위치가 재배치되어 예측성이 낮다.
   모바일은 1열이어서 길이 문제도 거의 그대로 남는다.

3. **SC3 유형별 섹션 구획**

   청약·전세·정책·임장 등 유형별 제목과 카드 그리드를 둔다. 주제 탐색은
   좋아지지만 현재 유형 필터와 역할이 겹치고, 유형이 8개라 빈/작은 섹션이
   많아질 수 있다. 명시적 섹션이라 접기 없이 전부 노출해야 한다.

## 전 탭 내부 스크롤 앱셸 비용 재산정

### 제안 구조

`#appTopbar`는 계속 sticky로 두고, 활성 비매물 패널에
`height: calc(100dvh - var(--props-panel-top, var(--app-top-h)))`,
`min-height: 0`, `overflow-y: auto`, `overscroll-behavior: contain`을
적용한다. 이름은 향후 `--panel-top`처럼 일반화하는 편이 맞지만, 1차
구현에서는 기존 변수를 먼저 재사용해 범위를 줄일 수 있다.

### B-53·B-98 반영 후 재산정

| 항목 | B-94 당시 | 현재 재산정 |
|---|---|---|
| topbar 높이 | 헤더/탭 좌표가 분리되어 상수 위험 | B-53 `--app-top-h` 실측·ResizeObserver 존재 |
| 패널 시작점 | 실제 y와 CSS 높이식 불일치 | B-98 `--props-panel-top` RAF+resize 실측 패턴 존재 |
| 매물 회귀 기준 | document 스크롤 잔여 | `document maxScroll=0` 기준과 테스트 선례 존재 |
| 구현 중심 | 앱셸 재작성 | 공통 활성 패널 규칙 + 탭별 예외 정리 |

**규모: M, 1~2일, 제품 코드 2~3파일 예상.** 핵심 CSS는 작지만 검증면이
넓다. 예상 영향은 `style.css` + `js/nav.js`, 변수명을 일반화하거나
공통 패널 래퍼가 필요할 때만 `index.html`이다. 개별 렌더 파일은 원칙상
수정하지 않는다.

**주요 리스크**

1. 모바일 `100dvh` 변화, iOS PWA PTR, 브라우저 주소창 축소.
2. 입력 중 키보드가 열린 자산/액션/수집함에서 포커스 항목 도달 여부.
3. 수집함 추가폼·Tiptap, 모달 `.mbody`, 내부 리스트 사이의 중첩 스크롤.
4. 대시보드 `data-goto` 전환 시 이전 패널의 scrollTop 보존/초기화 정책.
5. 매물 지도/목록 모드의 기존 fixed·sticky·overscroll 회귀.

따라서 UI 형태 변경과 스크롤 통일을 한 커밋에 섞지 말고,
① 공통 앱셸 → ② 액션 → ③ 수집함 순으로 독립 검증하는 편이 안전하다.
