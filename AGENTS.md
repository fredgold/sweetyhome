# AGENTS.md — 스위티홈 (SweetyHome)

> 이 레포에서 작업하는 모든 AI 코딩 에이전트(Codex 등)를 위한 규칙이다.
> **프로젝트의 전체 맥락·기술 스택·데이터 모델·불변 제약은 `CLAUDE.md`가 마스터 문서(SSOT)다.**
> **모든 작업·리뷰·커밋 전에 `CLAUDE.md` → `AGENTS.md` → `HANDOFF.md` 순서로 반드시 읽어라. `HANDOFF.md`가 없으면 사람에게 알리고 현재 상태를 확인한 뒤 진행하라.**
> 변경 이력은 `HISTORY.md`(날짜별 append 로그)에 있다.
> 이 파일은 `CLAUDE.md`의 제약을 다시 적지 않는다. 대신 (1) 여러 에이전트가 함께 작업할 때의 핸드오프 규칙과 (2) 자주 실수하는 지점만 짚는다.

## 문서 우선순위
1. `CLAUDE.md` — 프로젝트 SSOT, 불변 제약·스택·데이터 모델
2. `AGENTS.md` — 다중 에이전트 실행·충돌 방지 규칙
3. `HANDOFF.md` — 가장 최근 작업 상태·커밋·검증·남은 일

세 문서가 충돌하면 `CLAUDE.md` → `AGENTS.md` → `HANDOFF.md` 순으로 따르고, 충돌 내용을 사람(테디)에게 알려라. `HISTORY.md`는 과거 맥락이 필요할 때만 해당 구간을 읽어 토큰 낭비를 막는다.

## 프로젝트 한 줄
테디 ♥ 연정의 결혼·서울 내집마련 준비 보드 웹앱. Vanilla JS + Vercel 서버리스. 빌드 도구 없음.

## 실제 프론트엔드 구조 (중요 — 오해 방지)
프론트는 **단일 파일이 아니다.** 기능별로 분리돼 있다:
- `index.html` — 화면 HTML 구조 + CSS/JS 로드 순서
- `style.css` — 앱 전체 스타일 (단일 CSS 파일)
- `js/` — 프론트 로직 13개 파일, 순서대로 로드:
  `utils.js`(공통 유틸) · `auth.js`(PIN 로그인·토큰) · `state.js`(스키마·기본값·Redis/localStorage 저장) · `nav.js`(탭·대시보드 렌더) · `actions.js`(액션) · `scraps-form.js` / `scraps-render.js` / `scraps-import.js`(수집함) · `assets.js`(자산) · `properties.js`(매물·지도·AI평가) · `profile.js`(프로필) · `ai.js`(AI 상담) · `boot.js`(초기 렌더·앱 시작)
- `api/*.js` — 서버리스 함수

→ **새 로직은 해당 기능의 js 파일에, 새 스타일은 `style.css`에 넣는다.** 모든 걸 `index.html`에 몰아넣지 마라. (일부 오래된 문서가 "단일 index.html"이라 적혀 있으면 무시하고 실제 구조를 따른다.)

## 다중 에이전트 핸드오프 규칙 (Codex ↔ Claude Code)
이 레포는 Claude Code와 Codex가 함께 작업한다. 충돌을 막기 위해:

1. **같은 파일을 두 에이전트가 동시에 편집하지 않는다.** 작업 시작 전 `git status`로 커밋 안 된 변경이 있는지 확인하라. 있으면 다른 에이전트가 작업 중일 수 있으니, 그 파일은 건드리지 말고 사람(테디)에게 물어라. **절대 커밋 안 된 남의 변경분 위에 덮어쓰지 마라.**
2. **핸드오프는 커밋 단위로.** 작업을 마치면 즉시 의미 있는 메시지로 커밋한다. 큰 작업은 단계마다 자주 커밋해, 다른 에이전트가 언제든 깨끗한 지점에서 이어받게 한다.
3. **멈출 때 상태를 남긴다.** 작업이 중간에 끊기면(토큰 소진 등) 마지막에 반드시 남긴다: "완료한 것 / 진행 중이던 것 / 다음에 할 것"을 3줄 이내로, 커밋 메시지나 `HISTORY.md`에.
4. **큰 작업은 시작 전 단계 체크리스트를 만든다.** 이어받는 에이전트가 어디까지 됐는지 파악하도록.
5. **작업 완료 후 사람이 확인할 검증 포인트를 알린다.** (예: "로컬에서 매물탭 열어 붙여넣기 되는지 확인해 주세요")

## 검증 표준

- **실기기 관문**: 한글 IME 조합·`env(safe-area-inset-*)`·실제 터치 키보드·PWA pull-to-refresh 등은 헤드리스 Playwright로 완전히 재현 불가능한 자동화 한계 항목이다. 이런 항목은 "사용자 실기기 확인"을 완료 조건으로 `HANDOFF.md`에 명시하고, 코드 리뷰·구조적 diff 확인으로 대체 검증한 범위와 확인 못한 범위를 명확히 구분해 보고한다(B-31 인수 사례 참고).
- **렌더 타이밍 변경 검증**: 디바운스·지연 렌더·비동기화를 도입할 때는 "지연 중 동기 상호작용" 케이스(연타 직후 즉시 Enter/단축키/붙여넣기 등 DOM 오프셋을 동기로 읽는 진입점)를 반드시 검증한다 — 지연된 렌더와 최신 raw 데이터가 어긋나 커서·선택범위가 깨질 수 있다(B-107 회귀 교훈, `utils.js`의 `ceFlushDebounced()`가 모든 동기 진입점 앞에서 예약된 렌더를 즉시 flush하는 패턴 참고).
- **CSS 변수 소비자 확인**: CSS 커스텀 프로퍼티(`--*`)를 새로 도입하거나 기존 소비 코드를 다른 체계로 교체할 때는, 값을 생산(`setProperty`)하는 JS 코드와 그 변수를 실제로 읽는 `style.css` 규칙이 항상 짝을 이루는지 변경 시점에 전수 grep으로 재확인한다 — 소비자 없는 생산 코드가 남으면 매 프레임 무의미한 DOM 측정·계산이 계속 실행된다(B-121에서 발견한 `--nav-h`/`--topbar-h`/`--overlay-top` 무소비 변수 교훈 — B-53에서 소비처를 다른 변수 체계로 교체했지만 생산 코드는 그대로 남아 있었음).
- **파일 락 프로토콜**: 작업 착수 직전과 커밋 직전 반드시 `git status`로 커밋 안 된 변경을 확인한다. 스테이징(`git add`)은 본인이 작업한 파일만 명시적으로 지정하고(`git add -A`/`git add .` 금지), 다른 에이전트의 미커밋 변경은 diff가 겹치지 않아도 절대 손대지도 스테이징하지도 않는다(B-84·B-118·B-31 등에서 실제로 반복 적용된 패턴).

## 자주 실수하는 지점 (CLAUDE.md에 상세, 여기선 요약)
- **저장 스키마를 깨지 마라.** 상태 키는 `sweetyhome` 하나. 새 필드는 `state.js`의 기본값 보정(DEFAULT/load 가드)에 추가해 기존 데이터가 안 깨지게.
- **AI 호출은 `/api/messages` 프록시만.** `api.anthropic.com` 직접 호출 금지. API 키·Client Secret·PIN·토큰을 코드에 하드코딩 절대 금지. 시크릿은 환경변수로만.
- **공개 SDK Client ID 예외.** 브라우저 SDK 사양상 노출이 필수이고 배포 도메인 제한이 걸린 공개 Client ID만 코드에 둘 수 있다. Secret은 절대 해당하지 않는다.
- **새 환경변수가 필요하면** 코드에 넣지 말고 "Vercel Settings에 이 env를 추가해 달라"고 사람에게 요청하라.
- **배포**: `git push` → Vercel 자동 배포. 별도 빌드 없음.
- **디자인 토큰 재사용**: 색 역할 골드=매물, 그린=자산, 보라=AI. 실제 값은 `style.css`에 있으니 새로 만들지 말고 읽어 써라. 한국어 UI, 큰 버튼, 초보 친화.
